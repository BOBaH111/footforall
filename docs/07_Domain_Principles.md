# Domain Principles

## Назначение

Данный документ описывает фундаментальные принципы проектирования предметной области Footforall.

Эти принципы являются обязательными для всех модулей системы и имеют более высокий приоритет, чем детали реализации, структура базы данных или API.

При возникновении архитектурных вопросов решение должно приниматься в пользу соблюдения принципов данного документа.

---

# DP-001. Матч — центральный агрегат системы

Матч является основной бизнес-сущностью платформы.

Все остальные сущности существуют только относительно конкретного матча.

Примеры:

- регистрация игрока;
- игровые события;
- протокол;
- рейтинги;
- статистика;
- финансовые операции;
- отзывы;
- достижения.

Без матча эти сущности не имеют самостоятельной ценности.

---

# DP-002. Каждый завершённый матч создаёт ценность

После завершения матча система автоматически обновляет цифровую историю всех участников.

В результате завершения матча:

- обновляется рейтинг игроков;
- обновляется рейтинг судьи;
- обновляется рейтинг площадки;
- обновляется статистика;
- начисляются достижения;
- формируется история матчей;
- создаются финансовые операции;
- становятся доступны оценки участников.

Матч является главным источником роста ценности платформы.

---

# DP-003. История матчей неизменяема

После подтверждения протокола история матча считается завершённой.

Данные завершённого матча не изменяются.

При необходимости исправления используются дополнительные события или корректирующие записи, а не изменение существующей истории.

---

# DP-004. Игровые события являются источником истины

Все игровые данные формируются на основании зарегистрированных событий матча.

Примеры событий:

- начало матча;
- начало тайма;
- гол;
- автогол;
- пенальти;
- жёлтая карточка;
- красная карточка;
- замена;
- окончание тайма;
- окончание матча.

События являются первичными данными системы.

---

# DP-005. Производные данные не являются источником истины

Рейтинг, статистика, достижения и аналитика являются производными от игровых событий.

Источником истины считаются только:

- матч;
- состав участников;
- игровые события;
- итоговый протокол.

Все остальные показатели могут быть пересчитаны.

---

# DP-006. Один пользователь — несколько ролей

Пользователь имеет единый аккаунт.

В рамках одного аккаунта пользователь может одновременно выполнять роли:

- игрок;
- судья;
- владелец площадки.

Роли независимы друг от друга.

---

# DP-007. Игровое амплуа определяется для каждого матча отдельно

Игровое амплуа не является постоянным свойством пользователя.

Во время регистрации на матч игрок выбирает позицию, на которой будет выступать именно в данном матче.

Это позволяет:

- корректно рассчитывать рейтинг;
- автоматически формировать команды;
- учитывать особенности конкретного матча.

---

# DP-008. Жизненный цикл матча управляется конечным автоматом

Матч может находиться только в одном состоянии одновременно.

Переход между состояниями возможен исключительно по установленным бизнес-правилам.

Произвольный переход между статусами запрещён.

---

# DP-009. Доменная модель независима от технологий

Бизнес-логика не должна зависеть от:

- NestJS;
- PostgreSQL;
- HTTP;
- REST API;
- Flutter;
- внешних сервисов.

Доменная модель должна оставаться самостоятельной и переносимой.

---

# DP-010. Каждое бизнес-правило должно быть проверяемым

Любое бизнес-правило должно иметь возможность автоматической проверки.

Основные правила жизненного цикла системы покрываются модульными тестами.

Изменение бизнес-логики без обновления тестов считается нарушением архитектуры проекта.

---

# DP-011. Автоматизация предпочтительнее ручного управления

Если бизнес-процесс может быть автоматически выполнен без потери достоверности данных, система должна выполнять его автоматически.

Приоритет имеют:

- автоматическое распределение команд;
- автоматический расчёт рейтингов;
- автоматическое обновление статистики;
- автоматические финансовые операции;
- автоматические уведомления.

---

# DP-012. Цифровая репутация пользователя является стратегической ценностью платформы

История матчей, рейтинг, статистика, достижения, дисциплина и отзывы формируют цифровую футбольную репутацию пользователя.

Ценность репутации увеличивается с каждым завершённым матчем и становится естественным фактором удержания пользователей внутри экосистемы Footforall.

---

# DP-013. Сначала предметная область — затем реализация

Разработка любого функционала выполняется в следующем порядке:

1. описание бизнес-логики;
2. описание бизнес-правил;
3. проектирование доменной модели;
4. проектирование базы данных;
5. проектирование API;
6. реализация;

---

# DP-014. Объективность важнее субъективности

Все ключевые показатели платформы должны формироваться на основе объективных игровых данных.

Приоритет имеют:

- игровые события;
- подтверждённый протокол;
- факты участия в матче;
- финансовые операции.

Субъективные оценки (отзывы, оценки судьи, оценки площадки) используются только там, где объективные данные отсутствуют, и никогда не должны оказывать определяющее влияние на игровые рейтинги.
8. автоматическое тестирование.

Изменение порядка допускается только в исключительных случаях.
