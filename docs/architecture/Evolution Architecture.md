# 09. Evolution Architecture

## Назначение

Evolution Architecture определяет единый подход к развитию доменных сущностей Footforall.

Каждая сущность системы развивается автоматически на основании объективных событий, происходящих внутри платформы.

Основным источником изменений является завершённый матч.

---

# Основной принцип

Ни одна сущность не должна самостоятельно вычислять собственную эволюцию.

Расчёт выполняется специализированными доменными сервисами (Engine), объединёнными в Evolution Pipeline.

---

# Evolution Pipeline

Evolution Pipeline является оркестратором процесса обновления состояния агрегата.

Pipeline:

- принимает агрегат;
- принимает объективные данные (Evidence);
- последовательно вызывает необходимые Engine;
- применяет результаты к агрегату.

Pipeline не содержит бизнес-формул.

Pipeline не выполняет вычислений.

Pipeline не хранит состояние.

---

# Engine

Каждый Engine отвечает только за один аспект развития агрегата.

Примеры:

- Statistics Engine
- Rating Engine
- Reputation Engine
- Achievement Engine
- Streak Engine

Каждый Engine:

- получает текущее состояние;
- получает объективные данные;
- возвращает новое состояние.

Engine не изменяет Aggregate Root напрямую.

---

# Aggregate Root

Aggregate Root является единственной точкой изменения состояния.

Изменение выполняется исключительно через методы агрегата.

Например:

applyStatistics()

applyRating()

applyReputation()

applyAchievements()

---

# Value Objects

Каждый вычисляемый аспект агрегата оформляется отдельным Value Object.

Например:

PlayerStatistics

PlayerRating

PlayerReputation

Такой подход позволяет:

- инкапсулировать бизнес-логику;
- независимо развивать модели;
- уменьшить связанность системы.

---

# Evidence

Каждый Pipeline принимает только объективные данные.

Источником данных является завершённый матч.

Для каждой роли используется собственная модель Evidence.

Например:

MatchPlayerEvidence

MatchRefereeEvidence

MatchVenueEvidence

Evidence является неизменяемым объектом.

---

# Общая схема

Completed Match

↓

Evidence

↓

Evolution Pipeline

↓

Engine 1

↓

Engine 2

↓

Engine N

↓

Aggregate Root

---

# Принципы

## Single Responsibility

Каждый Engine отвечает только за одну бизнес-задачу.

---

## Open / Closed Principle

Новые Engine могут быть добавлены без изменения существующих.

---

## Immutability

Value Objects являются неизменяемыми.

Engine возвращают новые экземпляры объектов.

---

## Extensibility

Архитектура должна позволять без изменения существующего кода добавлять:

- новые виды рейтингов;
- новые виды статистики;
- достижения;
- серии;
- сезонные показатели;
- дополнительные показатели эффективности.

---

# Применение

Данный подход используется для всех ролей системы.

Например:

PlayerEvolutionPipeline

RefereeEvolutionPipeline

VenueEvolutionPipeline

Каждая роль имеет собственный Pipeline и собственные Engine.

Архитектура Pipeline является общей, однако бизнес-логика каждой роли полностью независима.

---

# Долгосрочная цель

Каждый завершённый матч автоматически обновляет цифровую историю всех участников.

Любая новая сущность платформы должна иметь возможность эволюционировать через собственный Evolution Pipeline без изменения существующей архитектуры.
